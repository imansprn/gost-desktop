# GOST Desktop — Fix Top 3 Issues

**Goal:** Fix the top 3 critical/important issues identified in the code review.

## Issues to Fix

### 1. C-2: Path Traversal in ConfigBuilder (❌ Critical)
**Problem:** `saveTemplate(type, name, content)` uses `"$name.json"` directly as a filename with no sanitization. A user could pass `../../etc/passwd` as name and write outside the intended directory.

**Files to change:**
- `ConfigBuilder.kt` — add name validation before file operations

**Concrete changes:**
- Add a `validateName(name: String): Boolean` or inline check against `^[a-zA-Z0-9_-]+$`
- Return early or throw `IllegalArgumentException` for invalid names
- Apply to both `saveTemplate()` and `buildServiceConfig()` where user-provided names are used

**Verification:**
- `saveTemplate("chains", "../../etc/passwd", "...")` should fail or reject
- `saveTemplate("chains", "valid-name", "...")` should work normally
- Existing tests still pass

### 2. C-4: Orphan Subprocesses in ProcessManager (❌ Critical)
**Problem:** GOST proxy processes may survive app shutdown. No `destroyForcibly()` on exit.

**Files to change:**
- `ProcessManager.kt` — add process tracking and shutdown hook

**Concrete changes:**
- Track all spawned `Process` objects in a `MutableList` or similar
- Register a JVM shutdown hook (`Runtime.getRuntime().addShutdownHook`) that calls `destroyForcibly()` on all tracked processes
- Ensure `start()` adds to tracking, `stop()` removes from tracking

**Verification:**
- On app exit, GOST process is killed (check with `ps aux | grep gost`)
- Manual `stop()` removes process from tracking
- Double-start doesn't leave orphan processes

### 3. I-6 + I-5: Refactor Singletons to Class-Based with Injectable Paths (⚠️ Important)
**Problem:** `ServiceRegistry` and `ConfigBuilder` are Kotlin `object` singletons with hardcoded `~/.gost-manager/` paths. Tests cannot isolate filesystem access.

**Files to change:**
- `ServiceRegistry.kt` — refactor from `object` to `class`
- `ConfigBuilder.kt` — refactor from `object` to `class`
- `ServiceRegistryTest.kt` — update for constructor injection
- `ConfigBuilderTest.kt` — update with `TemporaryFolder` or temp dir
- All callers of these singletons — inject instances instead of direct object references
- `ServiceWizardDraftStore.kt` — also has same hardcoded path issue (I-9), fix similarly

**Concrete changes for each:**
1. `ServiceRegistry` class: constructor takes `dataDir: String` (or `Path`). Keep a top-level instance for app usage if needed.
2. `ConfigBuilder` class: constructor takes `baseDir: String`. Same pattern.
3. `ServiceWizardDraftStore` class: constructor takes `filePath: String`.
4. Tests: `@Before` creates a temp directory, `@After` cleans up. Construct class instances with temp path.
5. Update DI points in `App.kt`, `ServiceFormScreenModel`, etc.

**Verification:**
- Tests pass in isolation (no writes to real `~/.gost-manager/`)
- App still creates configs in the right directory (`~/.gost-manager/`)
- No compilation errors from changing `ServiceRegistry.method()` to `instance.method()`
